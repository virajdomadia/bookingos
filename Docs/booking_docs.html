01 — product requirements document

What we're
building.

A white-label appointment booking platform. One codebase, many business clients. Each gets their own subdomain, branding, and isolated data.

💡
One-sentence pitch
The booking infrastructure layer small Indian service businesses pay ₹1,500–5,000/month for — delivered as a one-time ₹8,000–20,000 setup with a self-hosted subdomain.
🌐
Platform: Web only — all three projects
All projects in this portfolio are web applications only. No React Native, no Expo, no app store. The public booking flow is mobile-responsive — it works on any phone browser at 375px — but there is no native mobile app. This distinction matters when pitching to clients: "works on any phone" (browser) is a feature, "has a native app" is a separate project.

§1 Problem Statement

Small service businesses in India manage appointments through WhatsApp, phone calls, and paper diaries. This creates double-bookings, no-shows without notice, and zero visibility for customers who want to book outside working hours.

Existing solutions charge ₹1,500–5,000/month, are not localised for Indian timezones, and require ongoing subscriptions. A one-time build with a setup fee undercuts all of them.

§3 Race Condition Handling FIXED

⚠
Original claim was wrong
v1.0 claimed "no double-bookings ever." Under heavy concurrent traffic, row locking alone is insufficient without the correct transaction isolation level. The correct claim: double-booking prevention using SELECT FOR UPDATE within a SERIALIZABLE transaction, validated by load testing.
// apps/api/src/routes/bookings.ts
await prisma.$transaction(
  async (tx) => {
    // SELECT FOR UPDATE + SKIP LOCKED = row-level lock
    const conflict = await tx.$queryRaw`
      SELECT id FROM "Booking"
      WHERE "tenantId" = ${tenantId}
        AND status NOT IN ('CANCELLED')
        AND "startsAt" < ${endsAt}
        AND "endsAt"   > ${startsAt}
      FOR UPDATE SKIP LOCKED
    `;
    if (conflict.length > 0) throw new Error('SLOT_TAKEN');
    return tx.booking.create({ data: bookingData });
  },
  { isolationLevel: 'Serializable' } // ← critical line
);
✓
Load test requirement before claiming production-ready
50 simultaneous POST requests to the same slot → exactly 1 booking created, 49 SLOT_TAKEN responses. Only after this passes can you claim double-booking prevention.

Logs in at /admin
Access token → React state. Refresh token → httpOnly cookie. Never localStorage.
2
Confirms or cancels a booking
Optimistic UI: badge updates immediately. API fires in background. On failure → badge reverts + error toast.
3
Session expires (15 min) ← handle silently
Next API call returns 401. Frontend silently calls POST /auth/refresh. If succeeds → retry request, user notices nothing. If fails → redirect to login.

07 — auth architecture

Auth without
NextAuth.

Custom JWT on Express. AuthContext in React. Axios interceptor for silent refresh. No NextAuth — no conflicts.

ℹ
Token strategy
Access token: JWT, 15-min expiry, stored in React state (memory only, never localStorage). Contains: { userId, tenantId, role }.
Refresh token: opaque UUID, 7-day expiry, stored in httpOnly Secure cookie. Rotated on every use.

§1 Express Auth Middleware

// apps/api/src/middleware/auth.ts
export const authMiddleware = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET);
    req.user     = payload;
    req.tenantId = payload.tenantId;
    // ← This line activates RLS for every query in this request
    await prisma.$executeRaw`
      SELECT set_config('app.tenant_id', ${payload.tenantId}, true)
    `;
    next();
  } catch (e) {
    return res.status(401).json({ error: 'Invalid or expired token' });
  }
};

§2 AuthContext — React State

// apps/web/src/context/auth.tsx
export function AuthProvider({ children }) {
  const [accessToken, setAccessToken] = useState<string | null>(null);
  const [user, setUser] = useState<User | null>(null);

  // On mount: try silent refresh — keeps user logged in after page reload
  useEffect(() => { silentRefresh().catch(() => {}); }, []);

  const silentRefresh = async () => {
    const res = await fetch(`${API_URL}/auth/refresh`,
      { method: 'POST', credentials: 'include' });
    if (res.ok) {
      const { accessToken, user } = await res.json();
      setAccessToken(accessToken);
      setUser(user);
    }
  };
}

§3 Axios Interceptor — Silent Refresh on 401

// apps/web/src/lib/api.ts
const api = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_URL,
  withCredentials: true, // sends httpOnly cookie on every request
});

api.interceptors.response.use(res => res, async error => {
  if (error.response?.status === 401 && !error.config._retry) {
    error.config._retry = true;
    const { data } = await axios.post('/auth/refresh',
                                          {}, { withCredentials: true });
    setAccessToken(data.accessToken); // update React state
    error.config.headers.Authorization = `Bearer ${data.accessToken}`;
    return api(error.config); // retry original request
  }
  if (error.response?.status === 401) router.push('/admin/login');
  return Promise.reject(error);
});
